# CellTrail 正式環境部署檢查清單

> 用途：把 CellTrail 從「本機開發」推上正式環境前，逐項確認。打勾才算完成。
> 前提認知：本機開發的設定（弱密鑰、匿名 admin、localhost CORS、簡易伺服器）**對正式環境都不安全**。不照這份清單走，等於把刑事證據系統裸奔上線。
> 對齊版本：2026-05-19。

---

## A. 安全設定 —— 最優先，沒做完不要上線

- [ ] **`SECRET_KEY` 換強密鑰**
  - 為什麼：JWT 用它簽章。本機 `.env` 目前是 `ld7rqj7k`（8 碼弱密鑰）；被猜到就能偽造任何人的登入 token、冒充 admin。
  - 怎麼做：`openssl rand -hex 32` 產生，填入正式環境的 `.env`。換 key 會讓所有現有登入失效（需重新登入），屬正常。

- [ ] **`AUTH_ENABLED=true`**
  - 為什麼：`false` 時任何請求都被當成匿名 admin（開發後門），等於完全沒有權限控管。

- [ ] **`CORS_ORIGINS` 改成正式前端網域**
  - 為什麼：目前是 `localhost:5501` 等白名單。要改成實際前端網址，並移除所有 localhost 來源。

- [ ] **停用開發帳號 `admin / admin123`**
  - 為什麼：弱密碼、且明碼寫在 `schema.sql`，是現成入口。上線後停用（`is_active=false`）或刪除。

- [ ] **`CIDadmin` 上線後立即改密碼**
  - 為什麼：預設密碼 `436910619` 明碼寫在 `schema.sql`，看過原始碼的人都知道。

- [ ] **確認 `.env` 不在版控**
  - 為什麼：`.env` 含 `SECRET_KEY`、Google API key、DB 密碼。確認 `.gitignore` 有擋；若曾誤 commit 過，要清 git 歷史。

- [ ] **全站走 HTTPS**
  - 為什麼：JWT 與分享連結 token 都在 HTTP header / URL 明文傳輸。沒有 HTTPS = 可被中間人攔截、冒用。

- [ ] **Google Maps API key 設限**
  - 為什麼：金鑰外流會被盜刷。在 Google Cloud Console 限制 HTTP referrer 與每日用量上限。

## B. 資料庫

- [ ] **DB 帳密換掉**
  - 為什麼：本機是 `celltrail / celltrail`，弱。

- [ ] **套用 schema 與 migration（順序）**
  ```
  schema.sql → migration_permissions.sql → migration_account_requests.sql → migration_share_links.sql
  ```
  - 註：`schema.sql` 自 2026-05-19 起已修正為「自足」—— 即使先 schema 後 migration 也能正確種出 admin 帳號（先前的順序陷阱已解除）。三個 migration 仍需各自套用。

- [ ] **資料庫 volume 持久化 + 備份策略**
  - 為什麼：`raw_traces` 是刑事證據資料，容器砍掉資料不能跟著消失。需設定持久化 volume 與定期備份。

- [ ] **`audit_logs` 表防竄改**
  - 為什麼：稽核日誌是證據完整性的核心。應用層設計為只 INSERT；DB 層建議再用權限限制 UPDATE / DELETE。

## C. 服務部署

- [ ] **後端不要用單一 `uvicorn` 裸跑**
  - 為什麼：開發是 `uvicorn app.main:app`。正式環境建議多 worker（uvicorn workers 或 gunicorn）+ 前面擋 nginx 反向代理。

- [ ] **`--reload` 不要開**
  - 開發與正式都不加（Python 3.13 macOS 有 spawn bug；正式環境也不需要）。

- [ ] **前端不要用 `python3 -m http.server`**
  - 為什麼：那是開發用簡易伺服器，無效能、無安全性。正式環境用 nginx 服務靜態檔。

- [ ] **Redis 建議啟用**
  - 為什麼：geocode 快取，大檔解析效能差很多。離線雖不致命但會明顯變慢。

- [ ] **確認 `frontend/api.js` 的 API base 指向正式後端**
  - 為什麼：前端要打得到正式環境的後端網址，不是 localhost。

## D. 功能面

- [ ] **填充 `cell_towers` 座標表**
  - 為什麼：純數字 cell_id 沒有地址、geocode 不到；有業者基地台座標表才定位得出來。目前表是空的（架構已就緒）。

- [x] **背景影片改 R2 CDN** —— 已完成（2026-05-19，login / register / change-password 三頁）

## E. 上線後驗收

- [ ] 用 `CIDadmin` 登入成功（確認 `AUTH_ENABLED=true` 生效）
- [ ] 不帶 token 打需驗證端點 → 確認被擋（回 401）
- [ ] 上傳一個真實歷程檔 → 地圖正確顯示點位
- [ ] 建立分享連結 → 免登入開得起來、30 分鐘後失效（回 410）
- [ ] `audit.html` 看得到上述操作的稽核紀錄

## 風險分級摘要

| 等級 | 項目 | 不做的後果 |
|---|---|---|
| 🔴 必做 | A 區全部（SECRET_KEY、AUTH、CORS、HTTPS、預設帳號） | 系統可被冒充 admin、證據資料外洩 |
| 🟠 重要 | B 區（DB 帳密、備份、audit 防竄改） | 證據資料遺失或可被質疑竄改 |
| 🟡 建議 | C 區（部署架構）、D 區（cell_towers） | 效能差、部分定位失敗，但不影響安全 |

> 對刑事偵查系統而言，A、B 區直接關係到「這份證據在法庭上站不站得住」，不是選配。
