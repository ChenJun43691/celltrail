## 1. 專案概述

**CellTrail** 是一套刑事偵查用的「基地台連線歷程」匯入與視覺化系統。

- **使用者**：刑事警察（Project owner：高雄市刑大冠鈞，chen95572295@gmail.com / e43691@kcg.gov.tw）。
- **解決什麼問題**：偵查員向電信業者調取嫌疑人的通聯／上網歷程後，拿到的是格式雜亂的 CSV／Excel／PDF。CellTrail 自動解析這些檔案、把基地台位置標到地圖上，讓偵查員能視覺化嫌疑人的移動軌跡。

### 核心流程

1. 偵查員拿到電信業者交付的歷程檔（CSV / Excel / PDF）。
2. 上傳到系統 → 後端解析（normalize）→ 寫入資料庫（或「臨時查看」模式不落 DB）。
3. 地圖視覺化：基地台位置 + 精度圓 + 方位角扇形（Google Maps 風格全螢幕 UI）。
4. 證據完整性保全：軟刪、稽核日誌（audit_logs）、SHA-256 證據鏈、方位角北方基準標註。

### 證據完整性

系統把「這份地圖在法庭上站不站得住」當成第一原則。證據完整性等級從專案初期的 4/10 提升到 **9/10**，做法包括：所有刪除都是軟刪（保留資料）、每個敏感操作都寫稽核日誌、上傳檔案落 SHA-256 雜湊、方位角的「北方基準」必須由人工標註並附書面證據。

## 3. 系統架構

### 三層架構

```
┌─────────────────────────────────────────────────┐
│  第 3 層  前端靜態頁  port 5501                   │  瀏覽器開的 HTML/JS
│           index.html / login.html / admin.html… │  Leaflet 地圖、所有 UI
├─────────────────────────────────────────────────┤
│  第 2 層  FastAPI 後端  port 8000                 │  解析、API、JWT 驗證
│           uvicorn app.main:app                   │  Swagger: /api/docs
├─────────────────────────────────────────────────┤
│  第 1 層  PostGIS 資料庫（Docker 容器）port 5432  │  資料、帳號、稽核
│           celltrail_db                            │  + Redis（選配）port 6379
└─────────────────────────────────────────────────┘
```

**上層依賴下層**，啟動順序固定：資料庫 → 後端 → 前端。前端是純靜態檔，所有資料都靠打後端 API 取得。

### 認證模型

- **`AUTH_ENABLED=false`**：完全關閉登入，任何請求都視為匿名 admin（`id=0`，不在 users 表）。單機開發方便用。
- **`AUTH_ENABLED=true`**：啟用 JWT 驗證，所有需驗證端點必須帶 `Authorization: Bearer <token>`。正式環境必須用這個。
- JWT 效期 12 小時（一個工作天）；`get_current_user_optional` 在 `AUTH_ENABLED=false` 時回傳匿名 admin。
- 登入端點 `POST /api/auth/login` 吃 **form-encoded**（OAuth2 標準），不是 JSON。

### 權限兩層

| 層級 | 欄位 | 級別 |
|---|---|---|
| 全域角色 | `users.role` | `admin` / `user` |
| 專案層權限 | `project_members.permission` | `owner` / `collaborator` / `viewer` |

- `owner`：可授權他人、刪除目標、PATCH 方位角基準；等同該案件的 admin。
- `collaborator`：可上傳資料、查看軌跡。
- `viewer`：唯讀（map-layers、evidence-report）。
- 共用守衛：`assert_project_access(user, project_id, min_permission)`（`backend/app/security.py`）。

## 4. 環境建置與啟動

### 前置需求

- macOS（開發機）、Docker Desktop、Python 3.13、homebrew。
- Docker daemon 不會開機自動啟動 —— 每次重開機要先 `open -a Docker`，等 menu bar 鯨魚圖示穩定。

### 啟動步驟（六步）

```bash
# ── Step 1：啟動 Docker Desktop ──
open -a Docker          # 等鯨魚圖示穩定；docker info 不報錯即就緒

# ── Step 2：啟動 PostGIS 容器 ──
docker compose -f infra/docker-compose.yml up -d db
docker compose -f infra/docker-compose.yml up -d redis   # Redis 選配
# 等 docker compose ps 顯示 (healthy) 才能跑下一步（太早套 SQL 會失敗）

# ── Step 3：套用資料庫 schema 與 migration ──
docker exec -i celltrail_db psql -U celltrail -d celltrail < backend/app/db/schema.sql
docker exec -i celltrail_db psql -U celltrail -d celltrail < backend/app/db/migration_permissions.sql
docker exec -i celltrail_db psql -U celltrail -d celltrail < backend/app/db/migration_account_requests.sql
docker exec -i celltrail_db psql -U celltrail -d celltrail < backend/app/db/migration_share_links.sql
# ⚠️ 套用順序見第 5 節「schema 套用順序陷阱」

# ── Step 4：啟動後端 ──
cd backend
source .venv/bin/activate
uvicorn app.main:app --host 127.0.0.1 --port 8000
#   Swagger UI：http://localhost:8000/api/docs
#   注意：故意不加 --reload（Python 3.13 macOS 有 spawn bug）

# ── Step 5：啟動前端 ──
cd frontend
python3 -m http.server 5501
#   登入頁：http://127.0.0.1:5501/login.html

# ── Step 6：登入測試 ──
# 用下方系統管理員帳號登入
```

## 5. 資料庫設計

PostGIS 16。應用表共 **10 張**（另有 PostGIS 系統表 `spatial_ref_sys`、`geometry_columns`、`geography_columns`）。

### 5.1 `users` — 使用者帳號

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | bigint PK | |
| username | text UNIQUE | 帳號 |
| password_hash | text | bcrypt |
| role | text | `admin` / `user` |
| real_name / unit / badge_number / email | text | 姓名／單位／警號／信箱 |
| is_active | boolean | 軟停用 |
| must_change_password | boolean | 首次登入須改密碼 |
| created_at / updated_at | timestamptz | |

### 5.2 `raw_traces` — 基地台連線紀錄（核心資料表）

每列代表一筆經清洗與定位後的基地台連線。

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | bigint PK | |
| project_id / target_id | text | 案件代號／偵查對象 |
| msisdn / imei | text | 門號／設備碼 |
| start_ts / end_ts | timestamptz | 連線起訖（台北時間 naïve，由 `_parse_ts` 統一加時區） |
| cell_id / cell_addr | text | 基地台 ID／地址 |
| sector_name / site_code / sector_id | text | 細胞名稱／站台碼／磁區 |
| azimuth | integer | 方位角（度） |
| azimuth_ref | text | 北方基準：`magnetic` / `true` / `unknown`（預設 unknown） |
| lat / lng / accuracy_m | double / int | 座標與精度 |
| geom | geometry | PostGIS 點位（geocode 成功才有值） |
| deleted_at / deleted_by / delete_reason | | 軟刪欄位（永不真刪） |

### 5.3 `audit_logs` — 稽核日誌（append-only 帳本）

永遠 INSERT、永不 UPDATE/DELETE。欄位：`ts, user_id, username, role, action, target_type, target_ref, project_id, ip, user_agent, details(jsonb), payload_hash, status_code, error_text`。

### 5.4 `evidence_files` — 證據檔案雜湊

每次上傳落一筆：`filename, ext, size_bytes, sha256_full, uploaded_by, rows_total/inserted/skipped`。

### 5.5 `carrier_profiles` — 電信業者欄名對照表

`carrier_name, variant_label, fingerprint, mapping_json(jsonb), is_default, is_active, llm_assisted…`。

### 5.6 `cell_towers` — 本地基地台座標表

`cell_id UNIQUE, lat, lng, carrier_name, source, imported_by`。geocode 前置查詢用。**目前表是空的**，需向業者取得座標 CSV 匯入。

### 5.7 `project_members` — 專案成員權限（P3）

`project_id, user_id, permission(owner/collaborator/viewer), expires_at, granted_by`，`UNIQUE(project_id, user_id)`。`expires_at IS NULL`＝永久授權。

### 5.8 `account_requests` — 帳號申請（P3）

訪客送申請：`username, real_name, unit, phone, status(pending/approved/rejected), reason, reviewed_by`。admin 在 `/api/account-requests` 核准後建立正式帳號。

### 5.9 `format_reports` — 格式回報（P6）

不支援格式的回報：`filename, headers(jsonb), diagnosis(jsonb), status(open/handled/rejected)`。**只存 headers + 診斷，不存原始檔內容**。

### 5.10 `share_links` — 分享連結（P7）

| 欄位 | 說明 |
|---|---|
| token | text UNIQUE，`secrets.token_urlsafe(24)`（≈192-bit 熵） |
| project_id | 分享的專案 |
| created_by | 建立者（匿名 admin 寫 NULL） |
| expires_at | 後端寫死 created_at + 30 分鐘（原 12 小時，已改） |
| revoked_at | 撤銷時間（NULL＝未撤銷） |
| last_used_at / use_count | 觀察被開過幾次 |

### Migration 檔與套用順序陷阱

`schema.sql` 建基礎表；以下三個 migration **不在** schema.sql，新環境要另外套：

| 檔案 | 內容 |
|---|---|
| `migration_permissions.sql` | 擴充 users 表（real_name/unit/badge_number/email/is_active/must_change_password）+ 建 `project_members` |
| `migration_account_requests.sql` | 建 `account_requests` |
| `migration_share_links.sql` | 建 `share_links`（P7） |

> ⚠️ **隱患（本文件作者於 2026-05-19 發現）**：`schema.sql` 結尾種入 admin 帳號的 `INSERT` 用到了 `real_name`、`is_active`、`must_change_password` 欄位，但這些欄位是 `migration_permissions.sql` 才加的。**在全新資料庫上若嚴格「先 schema 後 migration」，種 admin 那兩行會因欄位不存在而失敗**（psql 預設不中止、會跳過）。
> **正確做法**：套完三個 migration 後**再跑一次 `schema.sql`**（冪等、`NOT EXISTS` 防重複），admin 帳號才會真正建出來。或手動補 INSERT。

## 7. 核心功能模組詳解

### 7.1 Ingest pipeline（檔案解析，`services/ingest.py`）

系統最複雜的部分。把雜亂的電信檔轉成乾淨的 `raw_traces`。

**三 phase 架構**（P6 重構，解決大檔逐筆 geocode 太慢的問題）：

1. **phase1**：normalize + 時間驗證（核心 `_normalize_row` / `_match_col_idx`）。
2. **phase2**：收集 unique `(cell_id, cell_addr)` → `geocode.lookup_bulk` 批次查詢（一次 SQL `ANY` + 一次 Redis `MGET`，剩餘 miss 才逐筆打 Google）。
3. **phase3**：組裝 records。

實測周蔓達 6769 筆檔案：403 秒 → 2.5 秒。

**關鍵子系統**：
- 兩層方言系統：全域 `_RAW2CANON`（負責 header 偵測投票）vs `_DIALECT_HEADER_MAPS`（dialect-specific 正確映射）。
- `_detect_dialect`：雙訊號偵測（headers 雙指紋 + ≥50% sample row）才認定方言，故意保守。
- `_match_col_idx`：兩階段欄位對應（Pass 1 精確、Pass 2 子字串備援），避免 silent 欄位錯位。

### 7.2 Geocode（地理編碼，`services/geocode.py`）

查詢優先序：① 本地 `cell_towers` 表 → ② Redis cache → ③ Google Geocoding → ④ OSM Nominatim 備援。Redis 全程包 try-catch，離線只是 cache miss。

### 7.3 權限系統（P3，`security.py`）

JWT + `get_current_user` / `require_admin` / `assert_project_access`。帳號申請流程：訪客 `POST /account-requests` → admin 核准 → 建帳號。

### 7.4 分享連結（P7，`share.py`）

- 安全模型：「持連結即可免登入檢視」，唯一防線是 token 不可猜測（192-bit 熵）。
- 純檢視：公開端點只回地圖 GeoJSON，無任何寫入／下載入口。
- 效期 **30 分鐘**（後端寫死，不接受呼叫端自訂，杜絕永久連結）；owner 可手動撤銷。
- 連結有效 = `revoked_at IS NULL AND expires_at > now()`；失效回 410，不存在回 404。
- 每次成功檢視 `use_count+1` 並寫一筆 `audit_logs`（`action=share_link.view`，含 IP）。
- 前端 `share.html` 為獨立精簡唯讀頁。

### 7.5 證據完整性

- `services/evidence.py`：上傳檔案落 SHA-256。
- `services/audit.py`：`write_audit()` helper，所有敏感操作寫稽核。
- 軟刪：`raw_traces.deleted_at`，永不真刪。
- `services/report.py` + `staticmap.py`：證據報告 PDF（含 OSM 靜態地圖截圖）。

### 7.6 格式回報三層機制（P6）

遇不支援格式時：① 解析失敗回 422 + diagnosis（`ParseDiagnosisError`）→ ② 前端手動欄位對應（`_apply_user_mapping`）→ ③ 回報 API 存進 `format_reports` 表供 admin 處理。

## 8. 關鍵設計決策（為什麼這樣做）

| 決策 | 為什麼 |
|---|---|
| 兩層方言系統 | 中華電信上網檔的 `起台/起址` 映射與 W1 既有規則衝突；直接改 W1 會壞掉 header 偵測投票，故拆成「偵測訊號」與「正確映射」兩套 |
| `_match_col_idx` 兩階段 | PDF header「細胞名稱」與「細胞」並存時，鬆散比對會讓 sector/cid 撞同一欄 → silent 資料汙染 |
| azimuth_ref 預設 unknown | 電信業者方位角的「北方基準」（磁北/真北）無統一規格，高雄磁偏角 -4~-5°、500m 距離差約 50m。系統不擅自推論，要 admin 附書面證據標註 |
| `_parse_ts` 拒絕時區後綴 | 系統假設全為 naïve 台北時間，由 `_parse_ts` 統一加時區，避免混入 UTC 造成 8 小時偏差 |
| popup 用 `azimuth != null` | 避免 `azimuth=0`（正北）被 truthy check 誤過濾 |
| 三 phase ingest | 大檔逐筆序列 geocode 每筆一次 round-trip，6769 筆要 6 分鐘；批次化後 2.5 秒 |
| 分享連結效期後端寫死 | 不開放呼叫端自訂，避免有人開出永久連結 |
| `share.py` 複製查詢不 import map.py | 讓 share 模組自足、不牽動既有 API |
| parse-only / parse-temp / upload 三種上傳 | 偵查員可先「看這檔能不能解」再決定是否建案 |

## 11. 已知問題與環境陷阱

1. **geocode 全敗會讓地圖空白**：`/map-layers` 的 SQL 有 `WHERE geom IS NOT NULL`，定位失敗的列被過濾。OSM 備援預設已開。
2. **Redis 沒跑不致命**：只是 cache miss + stats 去重失效。
3. **macOS Docker daemon 不自動啟動**：重開機要 `open -a Docker`。
4. **schema 套用 timing**：容器 `health: starting` 時套 SQL 會失敗，要等 `(healthy)`。
5. **schema.sql 種 admin 的欄位順序隱患**：見第 5 節 —— 全新 DB 要「schema → 三 migration → 再跑一次 schema」。
6. **PDF / 大檔首次解析慢**：cache 全 miss 時受 Google/OSM 節流限制，`上傳中…` 期間勿重複點。
7. **含中文路徑**：repo 路徑與樣本檔名含中文，shell 命令必須 quote。
8. **uvicorn 不要加 `--reload`**：Python 3.13 macOS 有 spawn bug。
